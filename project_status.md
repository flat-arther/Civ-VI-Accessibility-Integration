## UI Manager Rework (branch: UIManagerRework)

The UI manager has been rewritten as a class-based widget framework. The four old files (`baseWidget.lua`, `widgetTemplates.lua`, `widgetTemplateHelpers.lua`, `UIScreenManager.lua`) are deleted. The new framework lives under `src/UI/uiManager/` and `src/UI/uiManager/helpers/` with `CAIWidget_*` / `CAIWidgetHelpers_*` / `CAIUIScreenManager` file naming for VFS uniqueness.

- Framework is complete: `UIWidget` / `ContainerWidget` / `ValueWidget` base classes plus 20 concrete widgets (Button, MenuItem, StaticText, Panel, Dialog, List, HorizontalList, SubMenu, Tree, TreeItem, Dropdown, Checkbox, Slider, EditBox, TabControl, Tab, TabPage, Table, GameView, InterfaceMode). Five helper modules (Navigation, Search, Tree, EditBox, DialogBuilder).
- Manager-level additions over the old model: multi-listener event system (`On/Off/Emit`), single-source-of-truth focus path with directional entry (Windows tab-stop semantics), `FocusKey` + `CaptureFocusKey` + `RestoreFocus` for rebuild-survival, `Refocus` and `Announce` for out-of-band re-speak, path pruning on destroy, hidden-ancestor input skip, `_focusSound` auto-played on focus enter, `Transparent` widget flag, PgUp/PgDn paging, same-letter type-to-find cycling.
- Speech follows the Windows one-line-per-widget model. Multi-line emission routes through `SpeakLines(lines, interrupt)` where only the first line interrupts; the rest queue.
- Docs: `docs/ui-manager.md` is the source of truth (19 sections — architecture, focus, speech, events, widgets, helpers, migration). LuaLS annotations in `src/ideHelpers.lua`.
- **Migrated screens (all verified in-game unless noted):** MainMenu CAI block, `InGameTopOptionsMenu_CAI`, `ResearchChooser_CAI`, `CivicsChooser_CAI`, `ProductionPanel_CAI`, `CivicsTree_CAI`, `TechTree_CAI`, `GovernmentScreen_CAI`, `NotificationPanel_CAI`, `CivilopediaScreen_CAI`, and `CityStates_CAI` use the new framework. `ReligionScreen_CAI` (pending in-game test) also uses the new framework (CreateWidget, On("activate"/"focus_enter"/...), CaptureFocusKey/RestoreFocus, FocusKey rows, Push with mirrored vanilla flow). The disposable test screen and `UI/test/` directory were deleted. Every other `*_CAI.lua` still uses the old API and must be ported per `docs/ui-manager.md` §16 (see Known Remaining Work).
- **InGameTopOptionsMenu migration notes:** main button list activates rows by calling `child:DoLeftClick()` on the underlying vanilla `GridButton` instead of mapping controls to specific action functions; children of `MainStack` are filtered by non-empty `GetID()` so unnamed spacer Containers are skipped. The information section (`DetailsBox`) and the mods section (`ModsInUse`) are read-only `AlwaysEdit` EditBoxes — `SetText` normalizes `[NEWLINE]` to `\n` so no manual newline splitting is needed for the version tooltip; the mods listing's vanilla single-space spacer instance is emitted as a real `\n` separator, but only when it sits between real entries (leading/trailing spacer instances are trimmed). Expansion include chain: the file follows `Civ6Common`'s `IsExpansion1Active`/`IsExpansion2Active` to include `Expansion1_InGameTopOptionsMenu` when an expansion is active — required so the expansion's `LateInitialize` registers the `ExpansionNewFeatures` eLClick callback that `DoLeftClick()` relies on.
- **Expand/Collapse silent flag + recursive collapse + ancestor auto-expand (2026-05-29).** `TreeItemWidget:Expand(silent)` / `Collapse(silent)` and `SubMenuWidget:Expand(silent)` / `Collapse(silent)` now take a `silent` flag that suppresses **both** the `expanded`/`collapsed` event and speech (use for seeding initial state or auto-expanding). `Collapse` always recursively collapses every descendant in place (silent, no events) so re-expanding shows one clean level. **Speech rule: only user-driven toggles speak.** `TreeItem` focus speech still announces expand/collapse state (value element) on every node as the user navigates, and a deliberate Left/Right/Enter toggle speaks the new state via `SpeakElements({"value"})` (the default non-silent path used by the Tree key handlers). Every automatic/programmatic expand/collapse passes `silent=true` (seeding via `ExpandAllTreeItems`, CivicsTree/TechTree tier seed + era re-expand listener, `BuildFocusPath` ancestor auto-expand, GovernmentScreen seed), so opening an all-expanded tree/article and jumping focus never spam "expanded" — that earlier spam was automatic expands speaking before the silent flag existed. `CAIWidgetRegistry.ApplyProps` no longer drops setter-less primitive props or warns — it direct-assigns any prop with no `Set<Name>` setter (function or primitive). `BuildFocusPath` auto-expands collapsed expandable ancestors of a focus target (silent `Expand(true)`) so a target buried in a collapsed node is reachable; the target keeps its own state, and only path ancestors (not siblings) are expanded, so a jump into a collapsed CivicsTree/TechTree era reveals just the focused tier. Manual `widget.IsExpanded = true` writes were replaced with `widget:Expand(true)` in `CivilopediaScreen_CAI.lua` (`ExpandAllTreeItems`), `CivicsTree_CAI.lua`, and `TechTree_CAI.lua`; `ProductionPanel_CAI.lua`'s category seed stays a direct write because items are added after the node is created (Expand no-ops on a childless leaf). Unmigrated old-API screens (`DiplomacyActionView_CAI.lua`, `UnitPanel_CAI.lua`) still write `IsExpanded` directly and will be converted on their own migration. Verified in-game.
- **Manager mirrors vanilla flow** (durable rule). CAI never drives push/pop on its own; it observes vanilla's screen state and uses `mgr:RemoveFromStack(id)` (not `Pop`) when vanilla closes/rebuilds a section. Escape is NOT bound at the manager level — vanilla owns Escape, CAI follows. MainMenu submenu teardown is driven by inspecting `m_currentOptions[i].isSelected` after vanilla's `ToggleOption` runs.

## Purpose

Civilization VI accessibility mod for blind players. Adds TTS and screen-reader navigation through Lua UI replacements and wrappers.

- Authors: FlatArther and Hamada
- Source entry: `src/CivViAccess.modinfo`
- Deploy model: `src/` is symlinked into the Civ VI mod folder
- Core rule: all TTS goes through `Speak()` from `caiUtils.lua`
- API notes and discovered behavior live in `docs/game-api.md`

## Recent Decisions

- Shared unit naming lives in `inGameHelpers_CAI.lua`; avoid duplicating formation/name formatting logic.
- `SubMenu` in `widgetTemplates.lua` is now an entered child container, not a spoken expanded/collapsed tree-style node.
- Unit build improvements are grouped into a dedicated CAI submenu under unit `SelectionActions`.
- Disabled build-improvement actions are hidden only when vanilla provides no failure reason.
- Carrier/unit-hosted aircraft details are shared through `inGameHelpers_CAI.lua`; unit flags still speak count/capacity only.
- Plot-target and interface-target speech should prefer shared helper/data paths over screen-local reconstruction.
- City banner growth/production reads must support both vanilla base controls and XP1/XP2 stat-row instance managers; expansion banners move those tooltips and meters off the top-level banner instance.
- City banner bucket layout is now intended as `1` identity/status, `2` growth/production, `3` religion, `4` diplomacy, `5` loyalty, `6` governor, `7` power.
- City panel parity check shows `CityPanel_CAI.lua` still only covers compact summary reads plus action/toggle access; it does not yet mirror vanilla overview breakdown tabs (district/building/wonder/trading-post lists, religion beliefs, amenities/housing source breakdowns, detailed growth math, production queue) or XP1/XP2 loyalty/governor/power tabs.
- Main `CityPanel` buckets were redone to use single-purpose helpers plus shared request buckets. Current main-panel mapping is summary/tilde = name, population, health, production, growth; `Shift+1` name+health; `Shift+2` production; `Shift+3` food growth; `Shift+4` border growth; `Shift+5` religion; `Shift+6` population + housing + buildings or loyalty depending on active expansion; `Shift+7` through `Shift+0` are the yield buckets.
- City panel border-growth speech now aims at city-panel growth-tile parity: it includes the target hex direction relative to the city plus stored culture, culture per turn, required culture, and current / next-turn meter percentages.
- City `SelectionActions` ordering is now explicit instead of relying on the CITY hotkey-category order. Dynamic strike actions stay at the top, and the separate manage-citizens / purchase-tile actions are merged into one combined `Manage city` action that toggles both available city-management lenses together.
- City `SelectionActions` now append dynamic ranged-strike rows for the selected local-player city center and any strike-capable districts belonging to that city. Visibility is driven by `CityManager.CanStartCommand(..., CityCommandTypes.RANGE_ATTACK)` so the centralized list only shows attacks that vanilla would currently allow.
- City `SelectionActions` also append dynamic WMD strike rows for supported district launches such as missile silos. Availability is driven by vanilla-style `CityCommandTypes.WMD_STRIKE` target checks per district plot and per weapon type, and launching mirrors the banner `OnICBMStrikeButtonClick(...)` flow while also jumping the CAI cursor to the firing district.
- City panel action-list parity now includes the expansion overview tabs: `CityPanel_CAI.lua` follows the base / XP1 / XP2 `CityPanel` include chain, exposes XP1/XP2 loyalty-governor overview access through `LuaEvents.CityPanel_ToggleOverviewLoyalty()`, and exposes XP2 power overview access through `LuaEvents.CityPanel_ToggleOverviewPower()` when the underlying expansion capability is active.
- City-panel `Change production` accessibility should no longer blindly toggle `ChangeProductionCheck` when the vanilla production context is hidden. CAI now checks `/InGame/ProductionPanel` visibility first and calls `LuaEvents.CityPanel_ProductionOpen()` directly when needed, preserving vanilla queue-first tab choice while avoiding checkbox-state mismatches that consumed the first activation.
- `ProductionPanel_CAI.lua` now mirrors the Civilopedia-style rebuild focus handling: capture the focused body row before clearing, rebuild, restore by stable CAI focus key when possible, and otherwise reset the active body to its first child. Queue move/remove overrides still take precedence through the explicit post-rebuild queue index path.
- `NotificationPanel_CAI.lua` migrated to the new UI manager: Tree of TreeItem leaves / group nodes (FocusKeys `notification:<id>` and `group:<typeName>`); vanilla `GetMessage()` is the short title and `GetSummary()` is the longer content text, so group nodes use the title and individual leaves (single or inside a group) use the content text; tooltips and the location/coords helpers (`GetNotificationCoords`, `LookAtNotification` wrap) are dropped entirely per durable decision; alerts speak the content text (`GetSummary`, falling back to the title only when summary is empty) routed through `ProcessIcons` before `Speak`; add/dismiss while open trigger a full rebuild via `mgr:CaptureFocusKey`/`mgr:RestoreFocus`; close path uses `mgr:RemoveFromStack(NOTIFICATION_CENTER_ID)`; Escape binding on the tree itself (CAI owns this screen — no vanilla panel mirrors it) uses `KeyUp` per `feedback_nonnav_keys_keyup`.

## Durable Rules

- Prefer live vanilla control state for visibility, disabled checks, and exposed localized text.
- Prefer `GetText()` / `GetToolTipString()` over manual `Locale.Lookup()` when vanilla already exposes the text.
- Use `Locale.Lookup()` only when no control exposes the text or when the string is CAI-owned.
- Cache references and ids only; always read live values when speaking state.
- Tree expand/collapse logic belongs in shared templates, not screen-specific code.
- Pushed transient lists, trees, and dialogs should inherit current stack priority and close cleanly through the UI manager.
- Avoid persistent mirrors for ambient HUD panels when hotkey/read-on-demand access is better.
- End-turn blockers belong with ActionPanel access, not the notification center.
- For Civ VI gameplay DB values in Lua, remember that `0` is truthy. Explicitly test boolean-like DB fields and stat thresholds.

## Diplomacy migration (implemented 2026-05-29, pending in-game test)

Both diplomacy screens rewritten onto the new UI manager. Durable architecture facts:

- **DealView rides the vanilla wildcard host.** `DiplomacyDealView.lua` ends with `include("DiplomacyDealView_", true)` (a wildcard that pulls in every `DiplomacyDealView_*` file), with `Initialize()` called *after* it. So `DiplomacyDealView_CAI.lua` is registered as an InGame `<ImportFiles>` File (like `NotificationPanel_CAI`), **not** a `ReplaceUIScript`; it must NOT `include("DiplomacyDealView")`, and it only reassigns globals (`OnShow`, `InputHandler`, `PopulatePlayer*Panel`, `UpdateDealStatus`, …) — vanilla's later `Initialize()` registers them as handlers, so no `ContextPtr:Set*Handler` calls. This keeps every vanilla/DLC `DiplomacyDealView_*` override (e.g. KublaiKhan_Vietnam `_MODE`) intact.
- **ActionView stays a `ReplaceUIScript`** (no wildcard in `DiplomacyActionView.lua`) but now re-includes the expansion variant vanilla would pick (`_Expansion2` / `_Expansion1` / base), mirroring GovernmentScreen — otherwise the base script wins and silently drops all XP1/XP2 diplomacy logic (Alliance/Emergency/World-Congress tabs, expansion war types) on the standard ruleset. Because it is a full replace, it DOES re-register handlers via `ContextPtr:Set*` (vanilla `Initialize()` ran during the `include`).
- **ActionView user-facing layout unchanged:** leaders `Tree` → per-leader intel sections (`TreeItem`) → content nodes; statement actions are a sibling `List` (with `SubMenu`s for grouped options). No TabControl. View modes (overview / conversation / cinema) are gated by `HiddenPredicate` on two sibling panels + explicit `SetFocus`, replacing the old three-panel `SetActivePanel` focus-poking. Cinema speaks the leader greeting (was silent).
- **Intel tabs: silent enumeration + lazy per-tab switching + persistent per-leader subtrees** (reworked 2026-05-30, pending in-game test). The DLC alliance/emergency/world-congress adders build their buttons through `CreateTabButton` (+ `GetTabAnchor` + `LoadNewContext`), not `AddIntelTab`, so we must walk the finished `ms_IntelPanel.IntelTabButtonStack` to see all tabs. `CaptureIntelTabs()` now only *enumerates* the visible buttons into `{ Button }` tables (no clicking) — the earlier click-through-every-tab-on-every-`AddIntelPanel` was the source of the duplicate UI-click on leader-select and action-press: `DoLeftClick` fires the tab button's own native press sound (the `Style`-level SoundData; `ShowPanel` itself is silent — the old "buttons have no SoundData" comment was wrong). A tab's header and panel are resolved **lazily on first `focus_enter`** of its section: `tab.Header = IntelHeader:GetText()` (after the panel is the shown one) and `tab.Panel = VisibleIntelPanelChild()`. The switch (`tab.Button:DoLeftClick()`) is gated on **live `button:IsSelected()`** (ShowPanel `SetSelected()`s the shown tab, and DLC anchors set `m_ButtonInstance`/`m_HeaderText` so this works for them too) — so re-entering the current tab (Shift+Tab back from actions, moving among a tab's own children, focusing Overview right after a select since vanilla `ShowOverviewPanel()`s it) fires no click; only an actual tab change clicks. Base-reader instances are still captured by the eager `Populate*`/`OnActivate*` wraps, which vanilla calls inside `AddIntelPanel` (not on click), so removing the click-storm doesn't starve them. **Persistence:** `BuildSelectedLeaderChildren` builds each leader's sections once and keeps them; re-selecting a built leader (same tab count) only re-binds the pooled/recycled tab `Button`s in place (mutating the shared `tab` tables the section closures hold) and returns without teardown — so expand state, populated content (readers freeze the leader's live state; a leader screen never spans a turn) and focus position survive switching away and back. A changed tab count (DLC tabs arriving) falls through to a full rebuild. `entry.IntelBuilt`/`entry.Tabs`/`entry.IsSelf` track this. `m_state.buildingIntel` still gates `DiploScene_RefreshTabs`/`RefreshOverviewRows` from rebuilding mid-build. **Test in-game:** (a) selecting a leader and pressing actions no longer double-clicks; (b) expand a tab on leader A, switch to B, return to A — land back inside the still-expanded tab; (c) switching tabs still clicks once and reads correctly, incl. DLC tabs.

- **DealView uses a `TabControl`** (My offer / Their offer pages). Each page contains its offers `Tree`, inventory `Tree`, and the deal-actions `List` — the actions live *inside* each page (after inventory) so a plain Tab from the inventory tree lands on them via ordinary in-page nav (a sibling list beside the TabControl is unreachable; the framework's Tab cycle keeps focus inside the strip/page, and a root-sibling `ReleaseTabAtBoundary` experiment was tried and reverted — the nav model isn't built for it). Actions are deal-wide, so both pages' lists mirror the same vanilla controls and are rebuilt together by `RebuildActionsList`; FocusKeys are side-suffixed (`diplo:deal:action:*:<side>`). On rebuild, if a list is focused the cursor is moved to its first child (the leader-line text item) *silently* — speech is owned by the announcement queue below. A demand hides the relevant side's offers tree via `HiddenPredicate`. - **Bug-fix pass 2 (implemented 2026-05-30, pending in-game test).** Follow-ups to pass 1, mostly UI-manager-level: (5) **Focus returns to the overview position, never the leaders tree, after a conversation** — the ActionView `SelectPlayer` OVERVIEW branch no longer calls `SyncSelectedLeaderRow` on conversation return. (6) **Suppress the actions-focus restore during conversation/cinema transitions, screen-side — NOT by silencing the API.** Vanilla `SelectPlayer` rebuilds the statement list (→ `RebuildActionsList`) and only *afterwards* calls `SetConversationMode`/`ApplyStatement` (see `DiplomacyActionView.lua:1818` then `:1825`, and `OnDiplomacyStatement:2812/2816`), so the actions list restored+spoke before the conversation list got focus — and when a picked Ask-For-Promise sub-option dissolved its submenu the restore fell to the *speaking* positional path and landed on Casus Belli. Fix: `m_state.suppressActionsFocus` (set for `CAI_CONVERSATION_MODE`/`CAI_CINEMA_MODE` around `orig`, reset after) makes `RebuildActionsList` pass a **nil capture** so every `RestoreFocus` no-ops for that pass; the conversation list / cinema line is then the only thing that speaks. (7) **`RestoreFocus` stops at collapsed expandables** — the positional `path` walk breaks at any node whose `IsExpanded == false` instead of descending, so a captured position inside a since-collapsed submenu/tree-item lands on the node rather than silently re-opening it. (8) **SubMenu mirrors TreeItem expand/collapse** — `SubMenuWidget:GetDefaultChild`/`GetEntryChild` now also require a remembered focus child (`_lastFocusedChild`/`_lastFocusedKey`), so a bare `Expand(true)` (BuildFocusPath ancestor open, seeding) no longer auto-enters; and `EnterFirstChild` is a no-op once expanded, so a Right/Enter bubbling from a child can't re-enter and yank focus back to the first child.
- **Bug-fix pass 3 (implemented 2026-05-31, pending in-game test).** Two open-into-conversation focus fixes. (9) **Open-into-conversation lands on the conversation list, not the hidden leaders tree.** An AI-initiated request (e.g. allow an embassy in your capital) runs vanilla `OnDiplomacyStatement` → `SetConversationMode` + `ApplyStatement` *before* `OnShow` (vanilla's own OnShow note confirms this), so by push time the conversation container is shown + its list populated but the overview panel is hidden. `PushRootFocusingSelected` only special-cased cinema and otherwise focused the *selected leader's row* — which sits inside the hidden `overviewPanel`. Root cause beyond this screen: `SetFocus`/`BuildFocusPath` only skip hidden widgets while *descending* to a leaf; they do **not** reject an explicit target whose *ancestor* is hidden (programmatic focus trusts the caller). So focusing the leader row drove focus straight into the hidden tree. Fix kept at the screen level (mirroring the cinema branch): `PushRootFocusingSelected` now checks `IsConversationActive()` and pushes with `focus = m_ui.conversationList` when the conversation is the live view, before the leader-row fallback. Did **not** add an ancestor-visibility guard to the manager (would change focus semantics for every screen). (10) **Conversation rebuilds always reset to the first item.** `RefreshConversationPanel` was doing `CaptureFocusKey` → `RestoreFocus`, which restored the *positional* slot of the reply just picked (conversation reply buttons carry no `FocusKey`, so the capture only matched by index). Fix: dropped the capture/restore; after `ClearChildren` it nils the list's `_lastFocusedKey`/`_lastFocusedChild` entry-descent cache so `FocusConversationIfReady`'s `SetFocus(m_ui.conversationList)` descends via `DefaultIndex` → first visible child. The existing guards (root must be top, list must be populated) still prevent stealing focus when a deal screen is open or the demand routed straight to the deal. **Test in-game:** (a) AI request conversation opens reading the response text + reply choices, not the leaders tree; (b) picking a reply that yields a follow-up response reads the new response first, then arrow down into the new choices; (c) cinema intro still lands on the cinema panel and opening the screen yourself still lands on the selected leader row.
- **Focus-model rules learned (durable):** `RestoreFocus` by FocusKey is *silent*; positional `path` restore and first-visible-child fallback both re-`SetFocus` and *re-speak* (the positional walk additionally stops at collapsed expandables). A screen that hands focus elsewhere right after a rebuild should pass a **nil capture** to skip the restore for that pass, not lean on a silent restore. So all rebuilt rows that should restore silently need a `FocusKey` (statement actions `diplo:action:*`, deal actions `diplo:deal:action:*`, offers `diplo:offer:*`, leaders `diplo:leader:*`, side buttons `diplo:deal:side:*`). View panels are `Transparent` (no bare "panel" / duplicate-title speech). Conversation focus is deferred until the list has content (driven from both `SetConversationMode` and `RefreshConversationPanel`, whichever runs second) so the label/choices are fresh and demands don't focus an empty list. Leader-row selection sets a `selectingFromRow` guard so `SelectPlayer` doesn't re-focus (double-speak) the row the user just navigated to; same-leader returns from a deal leave focus on the action (e.g. Make Deal) rather than jumping to the leader.

## DiplomacyRibbon migration (implemented 2026-06-05, pending in-game test)
