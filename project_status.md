## UI Manager Rework (branch: UIManagerRework)

The UI manager has been rewritten as a class-based widget framework. The four old files (`baseWidget.lua`, `widgetTemplates.lua`, `widgetTemplateHelpers.lua`, `UIScreenManager.lua`) are deleted. The new framework lives under `src/UI/uiManager/` and `src/UI/uiManager/helpers/` with `CAIWidget_*` / `CAIWidgetHelpers_*` / `CAIUIScreenManager` file naming for VFS uniqueness.

- Framework is complete: `UIWidget` / `ContainerWidget` / `ValueWidget` base classes plus 20 concrete widgets (Button, MenuItem, StaticText, Panel, Dialog, List, HorizontalList, SubMenu, Tree, TreeItem, Dropdown, Checkbox, Slider, EditBox, TabControl, Tab, TabPage, Table, GameView, InterfaceMode). Five helper modules (Navigation, Search, Tree, EditBox, DialogBuilder).
- Manager-level additions over the old model: multi-listener event system (`On/Off/Emit`), single-source-of-truth focus path with directional entry (Windows tab-stop semantics), `FocusKey` + `CaptureFocusKey` + `RestoreFocus` for rebuild-survival, `Refocus` and `Announce` for out-of-band re-speak, path pruning on destroy, hidden-ancestor input skip, `_focusSound` auto-played on focus enter, `Transparent` widget flag, PgUp/PgDn paging, same-letter type-to-find cycling.
- Speech follows the Windows one-line-per-widget model. Multi-line emission routes through `SpeakLines(lines, interrupt)` where only the first line interrupts; the rest queue.
- Docs: `docs/ui-manager.md` is the source of truth (19 sections — architecture, focus, speech, events, widgets, helpers, migration). LuaLS annotations in `src/ideHelpers.lua`.
- **Migrated screens (all verified in-game unless noted):** MainMenu CAI block, `InGameTopOptionsMenu_CAI`, `ResearchChooser_CAI`, `CivicsChooser_CAI`, `ProductionPanel_CAI`, `CivicsTree_CAI`, `TechTree_CAI`, `GovernmentScreen_CAI`, `NotificationPanel_CAI`, `CivilopediaScreen_CAI`, and `CityStates_CAI` use the new framework. `ReligionScreen_CAI` (pending in-game test) also uses the new framework (CreateWidget, On("activate"/"focus_enter"/...), CaptureFocusKey/RestoreFocus, FocusKey rows, Push with mirrored vanilla flow). The disposable test screen and `UI/test/` directory were deleted. Every other `*_CAI.lua` still uses the old API and must be ported per `docs/ui-manager.md` §16 (see Known Remaining Work).
- **InGameTopOptionsMenu migration notes:** main button list activates rows by calling `child:DoLeftClick()` on the underlying vanilla `GridButton` instead of mapping controls to specific action functions; children of `MainStack` are filtered by non-empty `GetID()` so unnamed spacer Containers are skipped. The information section (`DetailsBox`) and the mods section (`ModsInUse`) are read-only `AlwaysEdit` EditBoxes — `SetText` normalizes `[NEWLINE]` to `\n` so no manual newline splitting is needed for the version tooltip; the mods listing's vanilla single-space spacer instance is emitted as a real `\n` separator, but only when it sits between real entries (leading/trailing spacer instances are trimmed). Expansion include chain: the file follows `Civ6Common`'s `IsExpansion1Active`/`IsExpansion2Active` to include `Expansion1_InGameTopOptionsMenu` when an expansion is active — required so the expansion's `LateInitialize` registers the `ExpansionNewFeatures` eLClick callback that `DoLeftClick()` relies on.
- **Expand/Collapse silent flag + recursive collapse + ancestor auto-expand (2026-05-29).** `TreeItemWidget:Expand(silent)` / `Collapse(silent)` and `SubMenuWidget:Expand(silent)` / `Collapse(silent)` now take a `silent` flag that suppresses **both** the `expanded`/`collapsed` event and speech (use for seeding initial state or auto-expanding). `Collapse` always recursively collapses every descendant in place (silent, no events) so re-expanding shows one clean level. **Speech rule: only user-driven toggles speak.** `TreeItem` focus speech still announces expand/collapse state (value element) on every node as the user navigates, and a deliberate Left/Right/Enter toggle speaks the new state via `SpeakElements({"value"})` (the default non-silent path used by the Tree key handlers). Every automatic/programmatic expand/collapse passes `silent=true` (seeding via `ExpandAllTreeItems`, CivicsTree/TechTree tier seed + era re-expand listener, `BuildFocusPath` ancestor auto-expand, GovernmentScreen seed), so opening an all-expanded tree/article and jumping focus never spam "expanded" — that earlier spam was automatic expands speaking before the silent flag existed. `CAIWidgetRegistry.ApplyProps` no longer drops setter-less primitive props or warns — it direct-assigns any prop with no `Set<Name>` setter (function or primitive). `BuildFocusPath` auto-expands collapsed expandable ancestors of a focus target (silent `Expand(true)`) so a target buried in a collapsed node is reachable; the target keeps its own state, and only path ancestors (not siblings) are expanded, so a jump into a collapsed CivicsTree/TechTree era reveals just the focused tier. Manual `widget.IsExpanded = true` writes were replaced with `widget:Expand(true)` in `CivilopediaScreen_CAI.lua` (`ExpandAllTreeItems`), `CivicsTree_CAI.lua`, and `TechTree_CAI.lua`; `ProductionPanel_CAI.lua`'s category seed stays a direct write because items are added after the node is created (Expand no-ops on a childless leaf). Unmigrated old-API screens (`DiplomacyActionView_CAI.lua`, `UnitPanel_CAI.lua`) still write `IsExpanded` directly and will be converted on their own migration. Verified in-game.
- **Manager mirrors vanilla flow** (durable rule). CAI never drives push/pop on its own; it observes vanilla's screen state and uses `mgr:RemoveFromStack(id)` (not `Pop`) when vanilla closes/rebuilds a section. For vanilla refreshes, refresh only the mirrored widget container that vanilla refreshed; do not pop/re-push the CAI root or rebuild unrelated siblings unless explicitly instructed by the user. Use the UI manager rebuild tools (`CaptureFocusKey` / `RestoreFocus`, or `PrepareFocus` when refreshing an unfocused container) to preserve focus locally. Escape is NOT bound at the manager level — vanilla owns Escape, CAI follows. MainMenu submenu teardown is driven by inspecting `m_currentOptions[i].isSelected` after vanilla's `ToggleOption` runs.
- **No fallback labels/tooltips when vanilla source is known** (durable rule, 2026-07-04). If vanilla Lua or XML clearly defines a control's text or tooltip, CAI should read that live control text/tooltip directly and let missing text surface as a bug. Do not add fallback labels/tooltips that mask broken or unexpectedly empty vanilla control state. The existing `GetText()` / `GetToolTipString()` wrappers returning `""` are enough.
- **EditBox AlwaysEdit highlight behavior (2026-06-08).** `HighlightOnEdit` on writable `AlwaysEdit` edit boxes is applied silently from the widget's `focus_enter` handler, so text is selected when focus lands on the box without speaking a separate edit-start line. Focus entry does not otherwise reposition the cursor. Read-only `AlwaysEdit` viewers preserve their cursor when focus leaves and returns. Silent `BeginEdit(true)` only activates the edit buffer; programmatic `SetText(...)` clears selection until focus entry reapplies it for writable boxes with `HighlightOnEdit`.
- **EditBox focus-leave commit flag (2026-07-04).** `EditBoxWidget` now exposes `CommitOnFocusLeave`, defaulting to `true` so existing writable `AlwaysEdit` boxes keep their prior auto-commit behavior when focus leaves. `UI/frontEnd/Multiplayer/StagingRoom.lua` chat input and `UI/inGame/ChatPanel_CAI.lua` chat input set `CommitOnFocusLeave = false`, so moving focus away from chat preserves the buffered text instead of committing or sending it; Enter remains the commit/send path.

## Purpose

Civilization VI accessibility mod for blind players. Adds TTS and screen-reader navigation through Lua UI replacements and wrappers.

- Authors: FlatArther and Hamada
- Source entry: `src/CivViAccess.modinfo`
- Deploy model: `src/` is symlinked into the Civ VI mod folder
- Core rule: all TTS goes through `Speak()` from `caiUtils.lua`; `ProcessIcons` runs inside `Speak` by default (third arg `processTokens`, true when nil, false bypasses)
- API notes and discovered behavior live in `docs/game-api.md`

## Durable Rules

- Prefer live vanilla control state for visibility, disabled checks, and exposed localized text.
- Prefer `GetText()` / `GetToolTipString()` over manual `Locale.Lookup()` when vanilla already exposes the text.
- Use `Locale.Lookup()` only when no control exposes the text or when the string is CAI-owned.
- Cache references and ids only; always read live values when speaking state.
- Tree expand/collapse logic belongs in shared templates, not screen-specific code.
- Pushed transient lists, trees, and dialogs should inherit current stack priority and close cleanly through the UI manager.
- Avoid persistent mirrors for ambient HUD panels when hotkey/read-on-demand access is better.
- End-turn blockers belong with ActionPanel access, not the notification center.
- For Civ VI gameplay DB values in Lua, remember that `0` is truthy. Explicitly test boolean-like DB fields and stat thresholds.

- **Bug-fix pass 3 (implemented 2026-05-31, pending in-game test).** Two open-into-conversation focus fixes. (9) **Open-into-conversation lands on the conversation list, not the hidden leaders tree.** An AI-initiated request (e.g. allow an embassy in your capital) runs vanilla `OnDiplomacyStatement` → `SetConversationMode` + `ApplyStatement` *before* `OnShow` (vanilla's own OnShow note confirms this), so by push time the conversation container is shown + its list populated but the overview panel is hidden. `PushRootFocusingSelected` only special-cased cinema and otherwise focused the *selected leader's row* — which sits inside the hidden `overviewPanel`. Root cause beyond this screen: `SetFocus`/`BuildFocusPath` only skip hidden widgets while *descending* to a leaf; they do **not** reject an explicit target whose *ancestor* is hidden (programmatic focus trusts the caller). So focusing the leader row drove focus straight into the hidden tree. Fix kept at the screen level (mirroring the cinema branch): `PushRootFocusingSelected` now checks `IsConversationActive()` and pushes with `focus = m_ui.conversationList` when the conversation is the live view, before the leader-row fallback. Did **not** add an ancestor-visibility guard to the manager (would change focus semantics for every screen). (10) **Conversation rebuilds always reset to the first item.** `RefreshConversationPanel` was doing `CaptureFocusKey` → `RestoreFocus`, which restored the *positional* slot of the reply just picked (conversation reply buttons carry no `FocusKey`, so the capture only matched by index). The existing guards (root must be top, list must be populated) still prevent stealing focus when a deal screen is open or the demand routed straight to the deal. **Test in-game:** (a) AI request conversation opens reading the response text + reply choices, not the leaders tree; (b) picking a reply that yields a follow-up response reads the new response first, then arrow down into the new choices; (c) cinema intro still lands on the cinema panel and opening the screen yourself still lands on the selected leader row.
- **Focus-model rules learned (durable):** `RestoreFocus` by FocusKey is *silent*; positional `path` restore and first-visible-child fallback both re-`SetFocus` and *re-speak* (the positional walk additionally stops at collapsed expandables). A screen that hands focus elsewhere right after a rebuild should pass a **nil capture** to skip the restore for that pass, not lean on a silent restore. So all rebuilt rows that should restore silently need a `FocusKey` (statement actions `diplo:action:*`, deal actions `diplo:deal:action:*`, offers `diplo:offer:*`, leaders `diplo:leader:*`, side buttons `diplo:deal:side:*`). View panels are `Transparent` (no bare "panel" / duplicate-title speech). Conversation focus is deferred until the list has content (driven from both `SetConversationMode` and `RefreshConversationPanel`, whichever runs second) so the label/choices are fresh and demands don't focus an empty list.
